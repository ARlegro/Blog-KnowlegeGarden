## 1.  프롬프트와 프롬프트 엔지니어링
프롬프트 = 사용자 입력

프롬프트는 사용자가 모델과 대화하거나 작업을 요청할 때 입력하는 지시문이다.
AI와 대화할 수 있는 수단으로
- 무엇을 만들고,
- 어떤 형식으로 출력하며,
- 어떤 관점과 제약을 따라야하는지 

이 모든 정보를 담아 **사람의 의도를 모델이 이해하기 좋은 구조로 만드는 역할**을 한다

*이를 설계하고 최적화하는 과정 = "프롬프트 엔지니어링"*
- 일반 사용자의 **요구를 모델이 해석할 수 있는 형태로 구조화**하고,
- 출력 품질을 높이기 위해 **프롬프트를 반복적으로 개선**하는 활동 전체를 포함 

> 일련의 설계 과정 

## 2.  왜 중요한가
>[!tip] 출력 품질에 큰 영향
>프롬프트는 결국 모델의 입력 설계 이기 때문에 출력품질에 직접적 영향을 준다.

효과적이고 구조화된 프롬프팅은 아래와 같은 이점을 가진다.
1. 가장 관련성이 높은 출력으로 이끔
2. 일관성 향상
	- 여러 번 반복해도 톤, 형식, 세부 구성 등이 고르게 유지됨
3. 제어력, 해석력 증가 
4. 오류와 편향 최소화 

같은 프롬프트라도 모델마다 다른 응답을 가질 수 있다. 따라서 모델에 따라 적합한 프롬프트를 파악하는 것이 중요하다. 




## 3.  프롬프트 Tip

>[!tip] 좋은 프롬프트는 아래의 틀에 기반해진다.
>역할 + 목표 + 입/출력 형식 + 제약조건 + 예시

1. 명확하고 정확하게 작성 : 작업, 개념을 충분히 설명하기 
2. 모델과 지속적으로 피드백 받기 : 프롬프트를 개선해나가는 순환 구조 
3. 불완전하고 현실적인 사용자 입력으로 테스트해보자 
4. 작업을 체계적으로 세분화해서 정보 전달하기 
5. 모델에게 불분면하거나 모호한 부분을 식별하도록 요청하기 
6. 실 사용자의 패턴을 고려하여 테스트하기 


## 4.  번외 - Dspy 프레임워크 


LLM을 함수처럼 안정적으로 호출하도록 만들어주는 “모델 최적화 프레임워크”

- 프롬프트를 일일이 작성하지 않아도 된다.
- 함수의 입력 ➡ 출력 구조로 LLM을 정의한다.
- 그 함수가 더 좋은 출력을 만들도록 자동으로 개선한다.


이런게 있다네... 프롬프트 짜는데 너무 피곤했는데 이거 사용하는 법 알면 편할 것 같네,.... 나중에 공부하자 

## 5.  프롬프팅 기술들 모음

❌프롬프팅은 단순 요청 방식이 아니다❌
✅프롬프팅은 문제를 어떤 방식으로 모델에게 가르치고 사고하게 만드는 전략의 집합✅


### 5.1.  제로샷 프롬프팅 

#### 5.1.1.  개념 
> 별도의 예시나 설명 없이, 지시문만으로 작업을 요청하는 방식

ex. "너는 여행 계획을 돕는 어시스턴트야. 사용자의 여행 계획 작성에 도움을 줘"

특징
- 많은 작업이 LLM의 능력 범위 내에 있다.
- 프롬프트가 짧고 단순해서 속도가 빠르다.
- 예제나 상세한 지침이 없다 ➡ 복잡한 규칙, 긴 추론 과정이 필요한 작업은 성능이 불안정해진다.

#### 5.1.2.  언제 좋을까❓
- 빠르게 아이디어나 초기 결과를 보고 싶을 때 
- 출력 형식 제약이 거의 필요 없는 작업 
- 테스트용 


### 5.2.  ICL 및 퓨샷 프롬프팅 

#### 5.2.1.  개념 
> 모델에게 직접 예시를 제공하여 문맥안에서 문제 해결 패턴을 학습하도록 유도하는 방식


- ICL = 문맥 학습 = In-Context-Learning 
	- 문맥 자체가 "학습 데이터"역할을 함 
	- 
- 퓨샷(Few-shot)
	- 소량의 잘 선정된 예시를 제공해 모델의 출력 품질을 높이는 기법
	- 모델에 관련된 예제나 시연의 작은 집합을 제공 
	- 문맥 내 예제를 활용해 성능을 향상시킨다.

예시
```text
사용자: “서울 2박 3일 여행 일정 만들어줘.”
예시 출력:
Day1: ~
Day2: ~

이번 사용자 입력에 대해 위와 같은 형식으로 답변해줘.
```
#### 5.2.2.  ✅장점
- 일정한 포맷을 안정적으로 유지 가능 
- 일관된 스타일, 구조, 품질 확보 가능 

#### 5.2.3.  💢단점
- 작업이 복잡해지면 필요한 예시가 늘어 복잡해짐


#### 5.2.4.  언제 사용하면 좋은가❓
> 출력 형식 고정 시 

효과적이긴 하지만 작업이 복잡할 떄는 도전에 직면 가능❓
이럴 떄는 cot 프롬프트와 같은 고급전략들이 점점 더 유용-> 모델의 논리적 추론 능력을 향상시킨다.

### 5.3.  역할 프롬프팅 
#### 5.3.1.  개념 
> 특정 역할이나 정체성을 가지고 작업을 수행하도록 LLM에 지시하는 방식 

ex. “너는 시니어 백엔드 면접관이야. 지원자의 답변을 기술 깊이 기준으로 평가해줘.”

#### 5.3.2.  효과
- 관점이 명확해짐
- 캐릭터 톤 유지에 유용
- 안정성이 올라감 

#### 5.3.3.  언제 사용하면 좋은가❓
- 전문가 관점의 분석이 필요할 때 
- 일관된 톤, 페르소나가 필요할 때 


### 5.4.  COT 프롬프팅 

COT = Chain Of Thought 

#### 5.4.1.  개념 
> LLM이 자신의 사고 과정을 명확히 설명하도록 유도해 결과의 정확성을 높이는 방법 

❌단순히 정답을 주게하는 것이 아님❌<br>
✅어떻게 생각해서 그 답을 도달했는지를 먼저 적게 만들고 그 다음 답을 말하도록 하는것✅

#### 5.4.2.  장점 
1. *복잡한 문제에서 정확도 향상* Cuz 모델이 스스로 오류 잡음 
	- 단계를 쪼개서 생각하기 때문에 모델이 스스로 오류를 감지하고 논리를 보완한다.
	- 일정 계획, 보정, 최적화 등에 매우 효과적
	  
2. *디버깅 시 유리*
	- 모델이 왜 이런 결과를 냈는지 알기 쉽다 
	- 어느 단계에서 추론이 꼬였는지 쉽게 파악 가능 
	  
3. *Reasoning 품질이 올라감*
	- 단순 요약이 아니라 근거,논리,조건 평가를 바탕으로 판단하게 한다.

#### 5.4.3.  예시

```bash
문제를 해결하기 위한 사고 과정을 단계별로 상세히 작성하고,
마지막 줄에만 '최종 답: <정답>' 형태로 출력해라.
```

```BASH 
13 × 27을 계산해줘.  
생각 과정을 단계별로 설명하고 마지막 줄에 최종 답만 적어줘.
```

```BASH
답을 바로 말하지 말고, 문제를 해결하기 위한 사고 과정을 
"단계별로 자세히" 작성해라.

다음 규칙을 따른다:

1) 문제를 작은 단계로 쪼개라  
2) 각 단계에서 왜 그 단계를 수행하는지 설명하라  
3) 중간 계산/추론 과정을 모두 보여라  
4) 마지막 줄에만 '최종 답: <정답>' 형태로 출력  
```

#### 5.4.4.  단점 💢

> 응답이 길어져 비용, 시간이 증가한다.


#### 5.4.5.  언제 사용하면 좋은가❓
- 복잡한 추론 문제
- 의사결정 과정이 중요한 업무 
- 시나리오 생성, 경로 계산, 코드 분석 등
