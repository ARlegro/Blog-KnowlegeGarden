### 0.1.  한국 IT업계 개발 방법론 현황 
세계 표준이랑 좀 다르다. 
한국은 빨리 빨리 문화 

그리고 기업 유형별로 다르다

- 스타트업
	- 85% 이상이 애자일 방법론 채택(스크럼, 린 스타트업)
- 대기업
	- 45% 애자일, 35% 하이브리드, 20% 폭포수 
- SI 기업
	- 40% 폭포수, 35% 애자일, 25% 하이브리드 
- 게임 회사
	- 45% 하이브리드, 30% 애자일, 25% 폭포수
- 핀테크
	- 90% 이상 애자일 


> 빨리 빨리 문화 ➡ 1주의 짧은 스프린트 주기 선호 (글로벌 평균 2~4주)


### 0.2.  한국 특유의 문화적 적응

1. 계층적 문화와의 충돌 → 전통적인 상명하복 문화가 애자일의 자율성과 충돌
2. 빨리빨리 문화 → 1주의 짧은 스프린트 주기 선호(글로벌 평균 2-4주)
3. 한국식 회의 문화 → 직접적 비판을 피하고 간접적 피드백 방식 채택

### 0.3.  주요 기업 사례

- 네이버 → 90% 애자일 채택
- 카카오 → 80% 애자일 → 한국식으로 수정된 스크럼
- 삼성전자 → 하드웨어(폭포수 모델), 소프트웨어(애자일로 전환 중)
- LG CNS → 고객 요구에 따라 70% 폭포수, 30% 애자일

### 0.4.  미래 전망

- DevOps 채택 가속화
- 젊은 개발자들의 애자일 선호로 점진적 변화
- AI 도구 통합으로 방법론 진화 중

## 1.  폭포수 모델(Waterfall Model)
### 1.1.  주요 단계

- 요구사항 분석 → 설계 → 구현 → 테스트 → 배포 → 유지보수

### 1.2.  장점 ✅

- 명확한 문서화 계획
- 예측 가능한 일정과 비용
- 대규모 팀 관리 용이
- 규제가 엄격한 산업에 적합

### 1.3.  단점 💢

- 변경사항 반영 어려움
- 늦은 피드백
- 리스크 발견 지연
- 고객 요구사항 변화 대응 어려움

### 1.4.  적합한 프로젝트

- 요구사항이 명확하고 변경이 적은 프로젝트
- 규제 준수가 중요한 프로젝트
- 하드웨어 개발이 포함된 프로젝트

### 1.5.  예시 1 : 대형 은행이 10년 이상 사용한 레거시 시스템 전면 교체 프로젝트
- 6개월간 요구사항 분석(모든 업무 프로세스 문서화)
- 4개월간 시스템 설계(아키텍처, DB설계 등)
- 12개월간 구현(모듈별 개발팀 배정)
- 6개월간 통합 테스트 및 안정화
- 단계적 배포(지점별 순차 적용)

### 1.6.  예시 2 : 정부 기관의 전자조달 시스템 신규 구축
- 3개월간 법령 검토 및 요구사항 정의
- 2개월간 상세 설계서 작성 승인
- 8개월간 개발 및 단위 테스트
- 3개월간 통합 테스트 및 보안 점검
- 시스템 운영 후 전면 오픈



## 2.  애자일 모델 (Agile Model)

### 2.1.  핵심 가치

- 프로세스와 도구보다 개인과 상호작용
- 포괄적인 문서보다는 작동하는 소프트웨어
- 계약 협상보다 고객과의 협력
- 계획에 따르기보다 변화에 대응

### 2.2.  주요 프레임워크

1. 스크럼 → 2~4주 스프린트, 일일 스탠드업 미팅, 회고
2. 칸반 → 시각적 보드, WIP 제한, 지속적 흐름
3. XP(익스트림 프로그래밍) → 페어 프로그래밍, TDD, 지속적 통합

### 2.3.  장점 ✅
- 빠른 피드백과 개선
- 변경사항에 유연한 대응
- 고객 만족도 향상
- 팀 협업과 소통 강화

### 2.4.  단점 💢
- 전체 일정 예측 어려움
- 문서화 부족 가능성
- 고객의 지속적 참여 필요
- 경험 많은 팀원 필요


### 2.5.  예시

#### 2.5.1.  음식 배달 중개 플랫폼 MVP 개발
- 2주 스프린트
- 스프린트 1-2 → 기본 주문 기능
- 스프린트 3-4 → 결제 시스템 통합
- 스프린트 5-6 → 실시간 배달 추적
- 매 스프린트마다 사용자 피드백 반영

#### 2.5.2.  온라인 교육 플랫폼 → 급증한 온라인 교육 수요 대응
- 1주 스프린트(빨리빨리 모드) → 매 주마다 결과물(기능 등) 도출
- 칸반 보드로 실시간 진행상황 공유
- 매일 15분 스탠드업 미팅
- 주 단위 배포 및 A/B 테스트

#### 2.5.3.  기존 은행 앱과 차별화된 간편 송금 서비스 개발
- 스크럼 + DevSecOps 결합
- 2주 스프린트, 일 2회 배포
- 자동화된 보안 테스트
- 규제 변경사항 즉시 반영

## 3.  하이브리드 모델(Hybrid Model)

### 3.1.  일반적인 구조

1. 계획/분석 단계 → 폭포수, 명확한 요구사항 정의
2. 개발/구현 단계 → 애자일, 반복적 개발
3. 배포/안정화 단계 → 폭포수, 체계적 검증

### 3.2.  장점 ✅

- 리스크 관리와 유연성의 균형
- 조직 문화에 맞는 점진적 전환
- 규제 요구사항과 혁신의 조화
- 다양한 이해관계자 만족

### 3.3.  단점 💢

- 복잡한 프로젝트 관리
- 명확한 가이드라인 필요
- 팀원들의 혼란 가능성
- 두 방법론의 단점만 상속하는 위험

## 4.  방법론 선택 가이드

| 프로젝트 특성        | 추천 방법론    | 이유          |
| -------------- | --------- | ----------- |
| 요구사항 명확, 변경 적음 | 폭포수       | 예측 가능성, 문서화 |
| 빠른 시장 진입 필요    | 애자일       | 신속한 MVP 출시  |
| 규제 준수 + 혁신 필요  | 하이브리드     | 균형잡힌 접근     |
| 스타트업, 불확실성 높음  | 애자일       | 피벗 용이성      |
| 대규모 SI 프로젝트    | 폭포수/하이브리드 | 관리 용이성      |
