### 0.1.  vs LangChain 표 정리
둘 다 LLM모델에 기반한 애플리케이션을 만드는데 도와주는 오픈소스 프레임워크이다.

> But **복잡한 사용자 의도 분기가 있는 서비스**는 LangGraph가 훨씬 유리하다 

| 항목      | LangChain          | LangGraph                           |
| ------- | ------------------ | ----------------------------------- |
| 구조      | Chain(순차)          | Graph(비선형)                          |
| 상태 관리   | 메시지 중심, 한정적        | 구조적 상태, reducer 기반                  |
| 도구 호출   | 단발성                | 반복적 tool-loop 안정적                   |
| 멀티 에이전트 | 비권장, 취약            | 설계 철학 자체가 multi-agent               |
| 실패 복원   | 없음                 | Checkpointer 기본 탑재                  |
| 적합한 문제  | 검색-요약 파이프라인, 단일 도구 | 여행 일정, 추천 재추천, 검색→평가→보정 같은 복잡한 의사결정 |


### 0.2.  목적의 차이 
*LangChain* #연결 
- LLM + 도구 호출 + 프롬프트 등을 “연결(Chain)
- LLM 기반 애플리케이션을 쉽게 만드는 도구 모음

*LangGraph*  #상태기반 
- 복잡한 multi-step 에이전트
- multi-agent workflow를 구조적으로 관리 
- **상태 기반** 에이전트 시스템을 만드는 프레임워크 



### 0.3.  구조의 차이 
*LangChain* #단방향 
- 정해진 순서로 흘러감 
- 순서가 명확한 작업에 최적화되었다 ✅

*LangGraph* #자유로운그래프 
- 상태가 A <-> B <->C 이렇게 되돌아거나 반복이 가능하다 
- 사용자 입력/조건에 따라 다음 노드가 달라지는 **동적 워크플로우** 구성 가능
- 대화형 시스템에 매우 강함 


### 0.4.  상태 관리 차이  ⭐ 핵심 
*LangChain*
- 메모리가 존재하지만 기능 한정적
- 단순히 대화 히스토리를 붙여주는 정도
- 여러 스탭 간의 구조화된 상태 관리가 어려움 

*LangGraph*
- state가 시스템 중심
- 모든 Node가 상태를 읽고/쓰고/업데이트 가능
- **장기 대화, 반복적 의사 결정, 복잡한 에이전트 플로우에 최적** 

### 0.5.  언제 사용하는게 좋을까❓

*LangChain*
- 간단한 검색 - 요약 파이프라인
- 단일 도구 호출 
- 명확한 순서의 프로세스 

*LangGraph*
- **장기 대화 기반의 에이전트** 
- Multi-Step 의사결정
- 실패 재시도 / 동적 흐름 필요한 
- 멀티 에이전트 협업
